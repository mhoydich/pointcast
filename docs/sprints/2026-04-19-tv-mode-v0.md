v0 of cross-platform big-screen PointCast. Fully custom HTML document (bypasses BlockLayout/BaseLayout — /tv has different chrome rules than the rest of the site). Key properties:

- **Presence**: reuses the existing `/api/presence` Durable Object WebSocket. `kind=tv` param added so the server can distinguish TV sessions from regular browsers if it wants to.

## Design decisions worth recording

- **Web-first, not native-first.** Mike said "on some levels we don't want to be confined to just apple tv os." The matrix brief to Manus makes the same point explicitly — casting + phone-as-controller gets us to every major TV platform today with zero store-submission friction. Native wrappers can come later if/when a specific platform's reach justifies the ops overhead.
- **Phone is always the controller.** Swipe/tap/QR routes all input through the phone, which (a) avoids the platform-specific remote API problem entirely, (b) lets us use existing Tezos wallet + localStorage session state, (c) keeps the TV truly ambient. Siri Remote, Roku remote, Fire TV remote — irrelevant. You don't need them.
- **QR code per slide.** Each slide's QR links to `/b/{id}` — so a viewer on the couch can tap it with their phone and land on the block with full interactivity. The QR is the handoff. No app install, no pairing flow, no custom protocol.
- **Reuse presence, don't re-architect.** `/tv` connects to the same `/api/presence` Durable Object as PresenceBar. This means a visitor on pointcast.xyz and a viewer on /tv are in the same presence set. That's the right default — we're not two services.
- **BlockLayout bypass is deliberate.** `/tv` wants different fonts (Lora + JetBrains only, no Outfit/Syne), different background (near-black instead of paper), different chrome (none). Fully custom HTML is less work than adding eight new props to BlockLayout.
