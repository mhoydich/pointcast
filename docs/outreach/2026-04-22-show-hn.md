## Short Draft

I built Sparrow, a static reader for PointCast that turns a personal broadcast into something closer to a tiny federated reading client.

The main experiment: saved lists and reading presence can move through Nostr instead of a central account system, while the site stays mostly static.

Receipts:

- Astro static output: 400+ routes, no client framework runtime.
- Every important surface has a machine-readable twin: `/sparrow.json`, `/sparrow/federation.json`, `/for-agents`, `/b/{id}.json`.
- Public saved lists are modeled as replaceable Nostr events so followed signers can see what each other is saving.
- The reader has a service-worker shell for offline return visits.
- PointCast itself is published as stable JSON "blocks" across nine channels.

Try the reader: https://pointcast.xyz/sparrow
Machine manifest: https://pointcast.xyz/sparrow.json
Agent/crawler map: https://pointcast.xyz/for-agents

## First Comment

OP here. A little more technical context.

PointCast started as a personal site, but the underlying shape became more like a broadcast protocol: every post is a typed JSON block with a stable ID, channel, type, timestamp, body, metadata, and permanent HTML/JSON routes. Sparrow is the reader skin on top of that archive.

The part I wanted to test is whether a small static reader can grow some social surface without becoming a SaaS app.

There are three layers:

1. Static read surface. `/sparrow` is an Astro-rendered reader over the block archive. The page is navigable by keyboard, has a service worker, and reads from static routes rather than a database-backed application shell.

2. Machine-readable surfaces. The reader exposes `/sparrow.json` and `/sparrow/federation.json`; the broader site exposes `/agents.json`, `/llms.txt`, RSS, JSON Feed, and per-block JSON. The goal is that humans, crawlers, and agents can all read the same archive without scraping styled HTML.

3. Nostr-adjacent saved-list federation. The idea is that a signer can publish a public saved list as a replaceable event. Other readers can follow those signers and get an ambient "what people I follow are saving" lane. Private cross-device state can be self-encrypted separately; public saved lists stay intentionally narrow.

The constraint was to keep the architecture small enough that it still feels like a personal site: static files, JSON manifests, a little client-side behavior, and optional federation. No global account, no feed algorithm, no central social graph.

Known rough edges: relay choice needs more testing, the UX around "follow a signer" is still early, and the Nostr event schema should probably be written up as a small RFC before it hardens.

Useful links:

- Reader: https://pointcast.xyz/sparrow
- Friends/federation view: https://pointcast.xyz/sparrow/friends
- Signal view: https://pointcast.xyz/sparrow/signals
- Machine manifest: https://pointcast.xyz/sparrow.json
- Federation manifest: https://pointcast.xyz/sparrow/federation.json
- Agent-readable site map: https://pointcast.xyz/for-agents

I am looking for blunt feedback on the protocol shape: what should be public, what should be private, what should be a replaceable event, and where this is overcomplicated.
